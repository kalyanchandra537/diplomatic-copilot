[1:00 - ARCHITECTURE]
(show assets/architecture_diagram.png)
Let's walk through the architecture of how this works behind the scenes.
First, we begin at the START node, leading to onboard_user, where we have a human-in-the-loop pause to collect the user profile and raw mentor text.
Once collected, this data flows into extraction_agent, which analyzes the text and runs the summarize_persona tool on our custom MCP server to compress the mentor's tone, frameworks, and vocabulary into a compact semantic embedding.
Next, the workflow transitions to get_dilemma, triggering another human-in-the-loop input, asking the user to submit their specific business challenge.
Before any LLM reasoning happens, the input is passed directly to the security_checkpoint node.
If any prompt injection or compliance violations are found, the flow immediately branches to compliance_warning to block the request.
Otherwise, the cleared input proceeds to the orchestrator agent, which uses the validate_dilemma_type and generate_diplomatic_opener tools from our MCP server, and delegates to the specialized execution_agent via an AgentTool to generate the final mentor-style advice.
Finally, format_output formats the advice and seals it with a green compliance badge before reaching the final_node.
